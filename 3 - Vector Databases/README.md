## 一、KNN

- **什么是 KNN？**
  - 它的全称是 **K-Nearest Neighbors**。简单来说，就是寻找离你最近的 `K` 个邻居。
- **具体步骤（图中的过程）：**
  - **Vectorize（向量化）**：利用 Embedding 模型，将知识库里的所有文档和用户的提问（Prompt）都转化成向量（空间中的点）。图中黄色的点就是提问，灰色的点是文档。
  - **Compute distances（计算距离）**：这是 KNN 的核心。计算黄色点到空间中**每一个**灰色点之间的距离（通常使用余弦相似度或欧式距离）。图中密密麻麻的虚线就代表了这些计算。
  - **Sort（排序）**：根据距离从近到远进行排列。
  - **Return closest elements（返回最近的元素）**：选择排在最前面的 `KK` 个文档。在第二张图中，那几个变红的点就是被选中的“最近邻”。

### 扩展性挑战

- **线性增长（Linear Growth）的陷阱**：
  - 图中横轴是“数据集大小”，纵轴是“计算时间”。两者呈直线关系，这意味着**数据量增加多少，计算耗时就增加多少**。
- **具体案例对比**：
  - **蓝点（小规模）**：如果你只有 1,000 个文档，每次搜索只需要做 1,000 次计算。计算机处理得非常快，你几乎感觉不到延迟。
  - **红点（大规模）**：如果你有 10 亿（1B）个文档，每搜一次，计算机都要老老实实地做 **10 亿次**距离计算。
- **核心矛盾**：
  - **计算量爆炸**：对于高频率、高并发的 RAG 系统，每次搜索都要扫描全库（暴力搜索）会导致响应极其缓慢，消耗巨大的 CPU 资源。
- **结论**：
  - **KNN 是“暴力搜索”**，虽然它能保证找到最精确的邻居，但它不具备扩展性。课件最后一句话提示：**“如果你想构建高性能的检索器，你需要更好的方法！”**（这通常是在引出 **ANN - 近似最近邻算法**，如 HNSW、FAISS 等）。

**KNN 是基准**：它通过遍历全库保证了 100% 的准确性。在你的数据集很小（例如几百条记录）时，直接用 KNN 是最简单、最稳妥的选择。

## 二、ANN

### 1. 什么是 ANN（近似最近邻）

- **显著更快速**：相比于检查每一个文档（KNN），ANN 只检查一小部分候选文档。
- **依赖额外的数据结构**：为了快，必须预先给数据建立“索引”（图中展示的复杂网络图）。
- **不保证绝对最优**：这是为了速度付出的代价。它找回的可能不是物理距离最近的那一个，但通常“足够接近”。

### 2. NSW（可导航小世界）图的构建

这页解释了如何把文档向量变成一个“可导航”的网络：

- **邻近图**：每个文档都是图中的一个**节点**。算法会将每个节点与其在空间中距离较近的邻居连接起来，形成**边**。
- **遍历能力**：由于节点之间有连接，我们可以沿着这些“边”在文档之间跳跃，从而快速在大规模空间中移动。

### 3. 查询入口点（Query Entry Point）

当用户提出问题（Query Vector，粉色点）时，检索开始：

- **起点**：算法在图中随机（或按某种规则）选取一个节点作为**初始候选点**。
- **逻辑**：现在我们站在绿圈标记的节点上，我们要去寻找离粉色点最近的节点。

### 4. 搜索过程（贪心算法）

这一页展示了检索是如何进行的：

- **跳跃（Hopping）**：从当前节点出发，检查它的所有邻居，看谁离粉色点更近。如果有更近的，就跳过去。
- **停止条件**：当你到达一个节点（图中青色点），发现它的所有邻居都比它自己离粉色点更远时，搜索停止。
- **局部最优**：这就是所谓的“贪心搜索”，你已经找到了在当前路径下的最佳位置。

### 5. 算法总结与局限性

- **可以找到多个邻居**：如果你需要 `K` 个结果，算法可以返回停止点附近的多个节点。
- **非全局最优**：由于它是沿着局部的边跳跃，可能会陷入“局部最优解”。也就是说，空间中可能存在另一个离粉色点更近的点，但由于没有路径连过去，算法没能发现它。
- **实战表现极佳**：虽然是“近似”，但在实际工程中，它的速度比 KNN 快成百上千倍，且找回的文档质量完全能满足大模型的需求。

## 三、分层改进 

### 1. 这页提出了 HNSW（Hierarchical Navigable Small World） 的核心定义：

- **改进 NSW**：HNSW 是对之前提到的 NSW（可导航小世界）算法的升级。
- **加速早期搜索**：它通过一种“分层”的结构，解决了在海量数据中搜索时起步阶段较慢的问题。
- **分层邻近图**：它不再只有一张平面的网，而是由多层相互关联的邻近图组成的。

### 2. 结构展示 (分层逻辑)

这页展示了 HNSW 的数据结构，我们可以通过**金字塔**或**行政地图**来理解：

- **Layer 3 (最顶层)**：只选取极少数（如 10 个）向量。这层连接非常稀疏，跨度极大。
- **Layer 2 (中间层)**：随机选取较多（如 100 个）向量。连接密集度适中。
- **Layer 1 (底层)**：包含**所有（1000 个）**向量。这是最完整的邻近图，连接非常紧密且精细。
- **逻辑**：每一层都是上一层的子集。这种结构类似于导航地图：全国地图（大城市） -> 省级地图（中小城市） -> 街道地图（所有建筑）。

### 3. 搜索过程

这页解释了如何在这种结构中寻找目标向量：

1. **第一步 (Layer 3)**：从顶层开始。因为点少跨度大，可以实现“**大步跳跃**”，迅速定位到目标所在的大致区域。
2. **第二步 (Layer 2)**：将顶层找到的最佳候选点作为进入第二层的入口。在这一层进行更细致的搜索。
3. **第三步 (Layer 1)**：进入底层。从上一层的优选点出发，在最密集的图中进行最终的微调，找到最精确的近邻。

- **优势**：这种“由粗到细”的策略极大减少了距离计算的次数。

### 4. 性能对比 (HNSW vs. KNN)

这页展示了为什么大系统必须用 HNSW：

- **KNN (Linear)**：计算时间随数据量**线性增长**。如果数据量增加 100 万倍，时间也增加 100 万倍。在处理数亿级数据时会彻底瘫痪。
- **HNSW (Logarithmic)**：计算时间随数据量呈现**对数级增长 (Logarithmic)**。**结论**：即使数据量达到十亿（Billions）级别，HNSW 依然能保持极快的响应速度（通常在毫秒级）。

### 5. ANN 总结

- **速度优势**：在大规模数据下，ANN（近似最近邻，如 HNSW）比 KNN 快得多。
- **牺牲精度换速度**：ANN 找到的是“非常接近”的文档，但不保证是“绝对第一”的那个点。但在 RAG 场景下，这微小的误差是可以接受的。
- **预计算成本**：构建这种“邻近图（索引）”在计算上是比较昂贵的（耗费 CPU 和内存），但它只需要在数据存入时计算一次（Pre-computed），之后的查询就会极快。

### 6. 总结：对 RAG 开发者的意义

1. **为什么我的 RAG 系统快？** 因为底层的向量数据库（如 Pinecone, Milvus, Weaviate）默认使用了 HNSW。
2. **索引构建 vs 查询**：当你把大量 PDF 存入向量数据库时会比较慢，因为系统在忙着建这些“层（Layers）”；但一旦存好，你提问时的检索几乎是瞬间完成的。
3. **扩展性**：有了 HNSW，你的 RAG 系统可以轻松应对从 1 万篇文档增加到 1 亿篇文档的挑战，而不会导致搜索变慢。

## 四、向量数据库

### 1. 为向量搜索而生

- **存储：** 向量数据库专门设计用于存储**高维向量**。这些向量是由 Embedding 模型生成的，代表了文字、图像或音频的含义。
- **搜索：** 它不仅是存数据，更重要的是它能通过 **ANN（近似最近邻）算法** 快速找到与查询向量最相似的数据。

### 2. 性能超越关系型数据库

- **传统数据库的局限：** 像 MySQL 或 PostgreSQL 这种传统的数据库是为精确匹配（比如查找 ID 或姓名）而设计的。
- **效率问题：** 如果在传统数据库中强行进行向量检索，它通常只能运行低效的 **KNN（暴力搜索）**，即把你的问题和库里每一条数据都对比一遍。当数据量达到百万或亿级时，传统数据库会变得极慢。
- **优势：** 向量数据库在处理这类相似性搜索时，速度要快出几个数量级。

### 3. 针对 ANN 搜索进行了优化

- **核心技术：** 向量数据库经过专门优化，能够构建 **HNSW 索引**（即你之前学过的分层小世界图算法）。
- **可扩展性：** 这种优化使得数据库在面对海量数据时依然能保持极快的响应速度（毫秒级），并且能够随着数据增长轻松扩展。

### 4. Weaviate：课程推荐工具

- **简介：** Weaviate 是一个非常流行的开源向量数据库。
- **地位：** 它是目前大模型生态中最主流的工具之一。
- **用途：** 在本课程的后续实验中，你将亲手操作 Weaviate 来存储文档向量并进行高效检索。

## 五、向量数据库的操作与工作流

### 1. 基本操作

1. **Database setup（数据库设置）**：
  - 安装并启动数据库服务，定义数据结构（Schema）。就像在传统数据库中建表一样，你需要告诉数据库你要存什么字段（如标题、内容、URL 等）。
2. **Load documents（加载文档）**：将你的原始资料（PDF、文本、网页等）导入系统中。
3. **Create sparse vectors for keyword search（创建用于关键词搜索的稀疏向量）**：系统会自动根据文本生成稀疏向量（通常是 BM25 算法），以便支持**精确匹配**搜索。
4. **Create dense vectors for semantic search（创建用于语义搜索的稠密向量）**：这是最关键的一步。通过 Embedding 模型，将每一段文本转换成代表其含义的高维稠密向量。
5. **Create HNSW index to power ANN algorithm（创建 HNSW 索引以支持 ANN 算法）**：为了让搜索变快，数据库会根据这些向量构建“分层小世界图”（HNSW）。这是为了在大规模数据中实现**毫秒级**的快速检索（近似最近邻搜索）。
6. **Run searches!（运行搜索！）**：一切准备就绪，你可以开始提问并获取最相关的文档了。

### 2. 完整工作流程

这一页将上面的具体操作高度概括为三个宏观阶段，展示了一个 RAG 系统后端建设的生命周期：

1. **Configure Database（配置数据库）**：
  - **动作**：设置环境、定义元数据字段、选择使用的 Embedding 模型。
  - **目的**：搭建好容纳数据的“容器”。
2. **Load and Index Data（加载与索引数据）**：
  - **动作**：这是最耗费计算资源的阶段。包括数据清洗、分片（Chunking）、生成向量（Embedding）、构建 HNSW 图。
  - **目的**：将原始文本变成计算机可以快速查找的“数字化地图”。这个过程通常被称为 **Ingestion（摄取）**。
3. **Perform Searches（执行搜索）**：
  - **动作**：接收用户提问，将其转化为向量，在索引中快速跳转找到邻居，返回结果。
  - **目的**：实现最终的 RAG 检索功能。

## 六、为什么要对文档进行切片

在构建 RAG 系统时，我们不会直接把一整本书或一份长达百页的 PDF 存入数据库，而是将其切分成一段段较小的文本块（Chunks）。课件从三个核心维度解释了这样做的必要性：

### 1. 词元限制—— 解决“装不下”的问题

- **物理限制：** 大语言模型都有**上下文窗口限制**。即使现在的模型窗口越来越大（如 128k 甚至更多），但单次 API 调用往往仍有 Token 上限。如果文档太长，模型根本读不完。
- **成本因素：** LLM 的计费通常是按 Token 数量计算的。发送不必要的长文档会极大增加运行成本。
- **切片的作用：** 确保每一块文本都足够小，能够轻松放入模型的输入窗口中，同时也节省了处理费用。

### 2. 提高相关性—— 解决“找得准”的问题

- **语义稀释：** 这是一个关键的数学问题。当你为一整本 200 页的书生成一个**向量（Embedding）**时，这个向量代表的是整本书的“平均含义”，它会变得非常模糊。
- **精准匹配：** 如果你把文档切成段落，每个段落的向量就会非常聚焦（例如：只讲“如何安装电池”）。当用户搜索特定问题时，向量数据库能更精确地匹配到那个具体的知识点，而不是找回一个包含该话题、但也包含几千句废话的大文档。
- **切片的作用：** 缩小语义范围，让搜索的精度从“某本书”提升到“某段话”。

### 3. 只给 LLM 发送相关的上下文 —— 解决“干扰少”的问题

- **去噪：** 如果用户问“这份合同的截止日期是哪天？”，你只把合同中关于“期限”的那一小段（Chunk）发给 LLM 即可。如果你把整份几十页的合同都发过去，LLM 可能会被其他条款干扰，产生幻觉或遗漏信息（这就是所谓的“Lost in the Middle”现象）。
- **效率：** 只提取最相关的 3-5 个片段给 LLM，能让它生成回答的速度更快，逻辑更清晰。
- **切片的作用：** 充当过滤器，剔除无关背景，只为大模型提供“精准的参考资料”。

## 七、如果不进行“切片”处理会发生什么

### 1. 不切片的索引方式

这张课件展示了一个极端的情况：将每一本完整的书直接转换成一个向量。

- **场景设定**：你的知识库里有 1,000 本书。
- **处理流程**：每一本书都被视为一个单一的整体，直接丢进嵌入模型。
- **结果**：你得到了 1,000 个向量。
- **核心动作**：图中展示了“一整本书”变成了一个固定的数字序列（向量），例如 [0.1, 0.4, 0.3, 0.2, 0.9]。
- **含义**：这意味着计算机认为一本书只有一个“坐标点”。

### 2. 这种做法存在的问题

五个核心痛点：

1. **过度压缩**：嵌入向量通常只有固定的长度（如 768 或 1536 维）。尝试用这么短的数字序列来表达几十万字的内容，就像试图用一句话总结一整部百科全书，会丢失海量的细节。
2. **无法表现细节**：一本书可能涵盖很多主题。如果第一章讲“苹果”，最后一章讲“斑马”，单一的向量无法同时精确地捕捉这两个截然不同的概念。
3. **产生“平均化”的表现**：数学上，整本书的向量会变成书中所有内容的“平均值”。这会导致语义变得模糊不清，缺乏针对性。
4. **导致搜索相关性极差**：当用户问一个非常具体的问题（例如“书里第 50 页提到的特定公式是什么？”）时，由于书的整体向量太模糊，检索器很难匹配到正确的那本书。
5. **撑爆大模型的内存**：即使检索器侥幸找到了正确的那本书，因为没有切片，系统必须把**整本书的文本**都发给大模型（LLM）。大多数 LLM 的上下文窗口（内存）装不下一整本书，即便装得下，处理速度也会极慢，且费用昂贵。

## 八、正确地对文档进行切片以及如何平衡切片的大小

### 1. 内容切片的层级

这张课件展示了将“一整本书”拆解为不同粒度（颗粒度）的可能性。

- **核心逻辑**：由于我们不能直接处理整本书，我们需要将其“破碎化”。
- **切片的三个常见层级**：
  - **按页**：较粗的粒度。通常包含多个主题，适合需要较多背景信息的场景。
  - **按段落**：最常用的粒度。通常一个段落围绕一个中心意思展开，语义比较聚焦。
  - **按句子**：极细的粒度。语义最精准，但由于缺乏上下文，往往难以表达完整的逻辑。

### 2. 平衡切片大小

为什么切片不能太大，也不能太小。这被称为 **“金刚波原则”或“恰到好处原则”**。

#### 1. 太大：章节级切片

- **语义稀释**：如果一个切片包含整个章节，里面可能聊了 10 个不同的主题。生成的向量会变成这些主题的“平均值”，导致搜索时不够精准（你想搜 A，但它混了 B 到 Z 的内容）。
- **占用空间**：大模型的“记性”是有限的。如果你给它扔几个巨大的切片，很快就会填满它的上下文窗口，导致它无法处理更多信息，或者增加计算成本（Token 费很贵）。

#### 2. 太小：词级切片

- **丢失语境**：如果切片只有几个词或一句话，模型就不知道这到底在说什么。比如“它坏了”，如果没有前后的句子，模型不知道“它”是指电脑还是手机。
- **降低检索相关性**：单个词的向量含义太单一，很难在搜索时匹配到复杂的、具有意图的用户提问。

#### 3. 恰到好处：最优切片

- **平衡点**：最优的切片应该既小到能够保持**语义聚焦**（只聊一个点），又大到能够保留**完整的逻辑和背景信息**。
- **实际操作建议**：在工程实践中，通常会将切片大小设定在 **300 到 500 个 Token** 左右，并且通常会设置 **10%-20% 的重叠度**，以确保切片之间语义的连贯性。

### 3. 总结：对 RAG 调优的启示

告诉我们：

1. **切片是艺术也是科学**：没有万能的切片大小，必须根据你的文档类型（是法律条文、代码还是小说）来决定。
2. **搜索精度 vs 上下文深度**：切片越**小**，搜索越**准**（Precision高），但 LLM 理解起来可能**断章取义**。切片越**大**，**背景越全**，但搜索可能**抓不住重点**。
3. **开发者的目标**：通过实验找到那个能让检索器准确找到答案、且能让 LLM 读懂逻辑的“甜蜜点”。

## 九、固定大小分片

### 1. 核心流程

图中的流程展示了如何处理一份长文档：

1. **Document（原始文档）**：一个完整的、长篇的文本文件。
2. **Chunking（分片处理）**：应用一个预设的规则，将文本机械地切开。
3. **Output（分片结果）**：文档被拆分成了多个独立的小块（Chunk 1, Chunk 2, Chunk 3...）。

### 2. 固定大小的含义

课件中给出了一个具体的参数：**每个分片 250 个字符**。

- **Chunk 1**：包含原文档的第 1 到第 250 个字符。
- **Chunk 2**：包含原文档的第 251 到第 500 个字符。
- **Chunk 3**：包含原文档的第 501 到第 750 个字符。

### 3. 这种方法的特点

#### **优点：**

- **简单直接**：实现起来非常容易，只需要按照字数（或 Token 数）进行计数并截断即可。
- **性能稳定**：计算开销极低，处理速度极快。
- **大小可控**：可以确保每一个分片都绝对不会超过大语言模型的上下文窗口限制。

#### **缺点—— 也是最严重的问题：**

- **破坏语义连贯性**：因为它只看字符数，不看内容。它可能会在**单词的中间**、**句子的中间**或者**段落的中间**强行切断。*例子*：原本一句话是“法国的首都是巴黎”，如果切片点正好在中间，Chunk 1 可能是“法国的首都是”，Chunk 2 可能是“巴黎”。这样 LLM 在读取 Chunk 1 时就拿不到完整答案。
- **丢失语境**：相关的逻辑信息可能被分到了不同的切片中，导致检索器搜到了其中一段，却丢失了必要的背景信息。

### 4. 实际应用中的改进

在现实的工程实践中，为了弥补这种“机械切分”带来的缺点，通常会引入 **“重叠（Overlap）”** 机制。例如，让相邻的两个分片有 50 个字符的重复内容，以确保语义不会在切分点被完全切断。

## 十、重叠分片

### 1. 核心概念：什么是重叠？

在上一页中，我们看到分片是紧挨着的（1-250，251-500）。而“重叠分片”则要求相邻的两个块之间有**一部分重复的内容**。

- **分片大小**：图中每个分片依然是 250 个字符。
- **重叠大小**：图中设置了 **25 个字符** 的重叠。
- **重叠比例**：25 / 250 = **10% 的重叠**。

### 2. 图解数据流

- **Chunk 1**：包含第 1 到 250 个字符。
- **Chunk 2**：从第 **226** 个字符开始，到 475 个字符结束。*注意*：第 226 到 250 这段文字，**既出现在 Chunk 1 的末尾，也出现在 Chunk 2 的开头**。
- **Chunk 3**：从第 451 个字符开始，到 700 结束（同样保留了上一块末尾的 25 个字符）。

### 3. 为什么要这样做？

- **减少语义断裂**：
  如果一个重要的信息（比如“北京是中国的首都”）刚好被切在了第 250 个字符处，固定分片会导致前半句在 Chunk 1，后半句在 Chunk 2。有了重叠，这句话至少在一个分片中能保持完整。
- **增强上下文完整性**：分片**中间**的单词两边都有丰富的背景。通过重叠，原本位于分片**边缘**（由于切分而变得孤立）的单词，现在会在两个分片中各出现一次。这样无论检索到哪一个，都能保留一定的上下文。
- **提高检索相关性**：
  由于语义更完整，向量数据库生成的 Embedding 会更精准，用户提问时更容易搜到正确的块。
- **存储代价**：
  这是它的副作用。因为重复存储了 10% 的数据，知识库的总体积会变大，存储和计算成本会略微增加。

## 十一、递归字符切分

**递归字符切分**，这是在构建 RAG 系统时最常用、也最推荐的一种文档切片方法。

### 1. 核心定义

- **在特定字符处切分**：它不是生硬地按字数（比如每 200 字）切开，而是寻找文本中自然的“分界线”，例如换行符（\n）、段落分隔符（\n\n）或句号（.）。
- **分片大小可变**：由于是按内容逻辑切分，每个分片的长度不一定完全相同，有的长一些，有的短一些。
- **兼顾文档结构**：这种方法最大的优势是它会尽量保持段落和句子的完整性，从而保留了文本原本的逻辑和语境。

### 2. 工作原理（为什么叫“递归”？）

虽然课件上只写了简单的描述，但在技术底层（如使用 LangChain 工具时），它的逻辑是这样的：

1. 它首先尝试用最宽泛的字符（如 **段落分隔符 \n\n**）来切分。
2. 如果切出来的块还是太大（超过了预设的最大长度），它会退而求其次，尝试用 **单换行符 \n** 进一步切分。
3. 如果还是太大，就尝试用**句号和空格**。
4. 这样层层递归，直到切出的每一块都符合规定的大小限制。

### 3. 视觉示例解析

图中展示了一段关于泰勒·斯威夫特（Taylor Swift）温哥华演唱会的新闻：

- **左侧（原始输入）**：是一段连续的文字，包含三个完整的意思。
- **右侧（切分结果）**：
  - **Chunk 1**：第一句话，关于演唱会的时间地点。
  - **Chunk 2**：第二句话，关于粉丝涌入和酒店需求。
  - **Chunk 3**：第三句话，关于酒店价格翻倍。

**对比意义**：如果使用“固定大小切分”，系统可能会在“caused hotel demand to”这里强行截断，把下一半放到另一个块里。而**递归切分**确保了每一块都是一个完整的语义单位。

### 4. 为什么要用这种方法？

- **提高检索精度**：当 LLM 查找背景资料时，如果拿到的资料是完整的句子或段落，它的理解会更准确。
- **减少语义断裂**：避免了“话只说一半”的情况，防止模型产生误解。
- **更聪明、更人性化**：它模仿了人类阅读文档的方式，优先考虑结构（段落 > 句子 > 词语）。

### 5. 总结

递归字符切分是 **“语义优先”** 的切片策略。它通过灵活的长度和对自然标点符号的尊重，解决了机械切分导致的语境丢失问题，是目前工业界处理非结构化文档（如 PDF、文章、合同）的首选方案。

## 十二、基于不同字符的切片策略

在 RAG（检索增强生成）系统中，如果只是机械地按照字数切分（比如每 500 字切一下），很容易把原本完整的逻辑给切断。这页课件告诉我们：**应该利用文档自身的“结构化语法”来进行更智能的切片。**

### 1. HTML 文档

- **策略：** 利用 HTML 的标签（Tags）进行切分，比如段落标签 <p> 或标题标签 <h1>、<h2>。
- **逻辑：** 在 HTML 中，标题和段落天然就代表了一个主题的开始和结束。
  - **例子（如右图红色和蓝色块）：** 将 `<h1>Document Title</h1>` 识别为一个块，或者将其与下方的描述连在一起。每个 `<p>...</p>` 标签内的内容是一个独立的语义单元。
- **优势：** 这样做可以确保**标题（Context）与其下的内容（Content）保持关联**，检索时大模型能清晰地知道这段话属于哪个章节。

### 2. Python 代码

- **策略：** 按照代码的逻辑结构（如函数定义 def 或类定义 class）进行切分。
- **逻辑：** 代码不是普通的文本，它具有极强的逻辑连贯性。如果把一个函数的上半部分放在 Chunk A，下半部分放在 Chunk B，大模型在读取时就无法理解这个函数的功能，甚至会导致语法报错。
  - **例子（如右图红色和蓝色块）：** 整个 `def function_one(): ...` 应该被视为一个完整的 Chunk（红色块）。整个 `class MyClass: ...` 以及它内部的 `__init__` 方法应该被尽量完整地保留（蓝色块）。
- **优势：** **保证了代码的可执行性和语义完整性**。检索出的代码片段可以直接被大模型理解、解释或运行。

### 3. 总结：为什么要这样做？

这种切片方式被称为 **“结构感知切片” (Structure-aware Chunking)**，其核心意义在于：

1. **保持语义完整性：** 确保每一个 Chunk 都是一个“有意义”的最小单位，而不是一段断章取义的文字。
2. **提高检索相关性：** 向量模型更容易识别出完整的函数或段落的意图，从而提高匹配精度。
3. **减少大模型的困惑：** 给 LLM 提供的是结构清晰、逻辑闭环的参考资料，它生成的回答会更准确，不容易产生幻觉。

## 十三、实施分片策略

### 1. 自主实现 vs. 使用现成库

这一条是在讨论**开发成本**和**灵活性**的平衡。

- **自主实现：** 最基础的分片方法（如带重叠的固定大小分片，Fixed size splitting with overlaps）逻辑非常简单。你可以直接用几行 Python 代码（比如字符串切片 `text[start:end]`）就能写出来。
  - **适用场景：** 当你的需求非常特殊，或者你不想引入额外的重量级依赖库时。
- **使用库：** 目前市面上有很多成熟的库（如 **LangChain** 的 TextSplitters 或 **LlamaIndex**）。
  - **优势：** 这些库处理得更精细。例如，它们可以确保不会在一个单词的中间切断，或者能够自动识别段落和标题的边界（即“递归字符切分”）。
  - **结论：** 课件指出，简单的分片逻辑无论是你自己写还是用库，实施起来都非常直接。

### 2. 元数据保留

这是工程实现中**最关键**但容易被初学者忽略的一点。

- **什么是元数据继承？** 当你把一份大文档（例如：`manual.pdf`）切成 100 个小块时，这些小块不能只是孤零零的文本。每一个小块都必须携带“母体”的信息，比如：文件名、作者、最后更新时间、文档分类、甚至访问权限。
- **位置信息：** 除了文档信息，还要记录这个小块在原文档中的具体位置。比如：它是从第几页切下来的？是哪一段？它的字符起始位置索引是多少？
- **为什么要保留元数据？**
  - **引用溯源：** 当 AI 回答用户问题时，系统可以利用元数据告诉用户：“这个答案参考自《用户手册》第 45 页”。
  - **元数据过滤：** 像我们之前学过的那样，如果你只想从“2024 年以后的文档”中搜索，系统需要通过这些继承下来的元数据来进行筛选。
  - **调试：** 方便开发者快速找到引起幻觉或错误信息的原始出处。

**千万别丢了元数据**。分片后的文本块必须包含溯源信息和原始文档属性，这样你的 RAG 系统才是可追踪、可过滤且专业的。

## 十四、语义分片

### 1. 核心对比：固定大小分片 vs. 语义分片

课件的前两张图通过一个例子展示了两种切片方式的区别：

- **左侧：固定大小分片** 做法： 机械地按照字符数限制切断。**问题：** 你可以看到图中，单词 "Maple" 被切成了两半（Map | le），"Canadian landscapes" 也被切断了。**后果：** 这种切法破坏了句子的完整性，导致语义支离破碎，模型在检索时很难理解这一块到底在说什么。
- **右侧：语义分片** 核心理念： **“基于意思的相似性进行分组，而不是靠随意的字符限制。”**结果： 关于“枫糖浆”的内容被归为一类（橙色），关于“加拿大景观”的内容被归为另一类（蓝色）。每一块都是完整的语义单元。

### 2. 语义分片的工作原理

详细拆解了语义分片具体是如何通过算法实现的。它不再是“数格子”，而是通过**数学向量**来“理解”内容。

#### 步骤拆解：

1. **逐句移动**：
   算法不是按字符处理，而是先将文档拆分成一个个完整的**句子**。
2. **向量化与对比**：将当前的文本块和下一句分别转化成**嵌入向量**。计算它们之间的**余弦距离**。简单来说，就是衡量这两部分在“意思”上离得有多远。
3. **检查阈值**：系统预设一个“相似度门槛”（阈值）。**如果距离低于阈值**（即意思很接近）：说明下一句还在聊同一个话题，就把这句话加入当前的文本块中。
4. **不同即切分**：**如果距离超过了阈值**：说明话题发生了转折（比如从聊“枫糖浆”变成了聊“山脉”），此时立即结束当前分片，并开启一个新的分片。

### 3. 为什么语义分片对 RAG 很重要

- **提高检索精度**：
  在 RAG 的检索阶段，用户问一个问题，系统去向量数据库找最相关的块。如果块里的内容语义高度聚焦，找回来的参考资料就会非常精准。
- **减少大模型的困惑**：
  由于分片是按意思切的，发给大模型的上下文是逻辑完整的，不会出现“话只说一半”的情况，从而显著减少模型幻觉（胡编乱造）。
- **自适应性**：
  它能自动适应不同长度的段落。一段长文字如果意思统一，它就是一个大块；如果话题跳跃快，它就会自动切成小块。

 **RAG 开发中，数据处理的质量直接决定了最后问答的效果。** 语义分片虽然比固定分片更消耗计算资源（因为需要多次调用 Embedding 模型），但它能提供质量更高的上下文，是构建高性能 RAG 系统的关键技术。

## 十五、优点和缺点

### 优点 —— 为什么它效果更好？

1. **遵循作者的思路：**固定大小的分片经常会在段落中间、甚至一句话中间强行切断。语义分片会观察语义的变化，确保同一个观点、同一个逻辑块被保留在同一个分片中。这使得分片内容更符合人类写作的逻辑。
2. **更智能的分片边界：**它不是机械地每 250 个字切一刀，而是寻找“语义断层”。这意味着分片的边界是根据内容自然形成的，避免了“断章取义”的问题。
3. **更高的召回率和精确率：**精确率： 检索到的内容更聚焦于用户的问题，杂音少。**召回率：** 因为意思完整的块更容易被匹配到，系统更不容易漏掉正确答案。最终结果是：大模型拿到的参考资料质量更高，回答更准确。

### 缺点 —— 它的代价是什么？

1. **计算成本昂贵：**固定大小分片几乎不消耗算力（只是字符串切片）。语义分片需要运行机器学习模型来理解每一句话，这需要大量的 CPU/GPU 资源。如果文档量巨大，处理速度会明显变慢。
2. **需要重复的向量计算：**为了找到最佳切分点，算法必须把文档中的**每一句话**都转化成向量，并不断计算相邻句子之间的相似度。相比之下，其他方法只需在分片完成后计算一次向量。语义分片在“分片前”就需要进行海量的预计算，这会增加 API 调用成本或本地推理时间。

## 十六、基于语言/大模型的分片策略

这是目前 RAG（检索增强生成）领域中**最智能、最精细**的一种分片方法。它不再依赖简单的字符计数或数学上的向量距离，而是直接利用大语言模型的**语义理解能力**来拆分文档。

### 1. 核心概念：让 AI 帮你切分

这种方法的操作逻辑是：**把原始文档发给 LLM，并给它下达指令，让它决定在哪里切分最合适。**

- **Prompt LLM to create chunks**：开发者编写提示词，告诉模型：“请阅读这段文字，并在语义发生转折的地方将其拆分成有意义的小块。”
- **Include instructions**：你可以在提示词中加入具体的约束。例如：“确保每一个分片都包含一个完整的概念”，“当新话题开始时进行分段”，“不要在列举项中间切断”。

### 2. 图解流程：从文档到概念块

观察底部的流程图：

1. **Document（文档）**：输入的长文本。
2. **LLM（大模型）**：模型充当“智能剪刀”，它不仅仅在数单词，而是在理解内容。
3. **Output（分片结果）**：**Chunk 1 (Concept A)**：这一块专门讲概念 A。**Chunk 2 (Concept B)**：这一块专门讲概念 B。**Chunk 3 (Concept C)**：这一块专门讲概念 C。**结果**：每一个 Chunk 都是一个**逻辑闭环**，没有任何多余或缺失的信息。

### 3. 为什么选择这种方式？

- **表现极佳**：由于 LLM 拥有最强的语义处理能力，这种方法产生的切片质量是最高的。它几乎不会出现“断章取义”或“语义截断”的问题。
- **经济可行性提高**：**过去**：用 LLM 来做分片非常昂贵且缓慢（因为分片是“预处理”阶段，数据量可能非常大）。**现在**：随着更便宜、更快速的模型（如 GPT-4o-mini, Claude Haiku）出现，大规模使用 AI 进行预处理的成本显著下降，使得这种“奢侈”的方法开始进入实用阶段。

### 4. 这种方法与其他分片的区别

- **固定大小分片**：像拿尺子量，每 10 厘米剪一刀，不管剪到了哪里。
- **语义分片（基于 Embedding）**：像拿数学公式算，看两句话的“数值距离”是否太远。
- **基于语言的分片**：像请一个**人类小秘书**帮你读一遍，然后用笔在纸上标出：“这里在聊新话题了，从这里切开吧”。

### 5.总结与启示：

这种策略代表了 RAG 的进化方向：**从“暴力切分”转向“理解后切分”**。

**优势**：检索精度最高，发给 LLM 的背景资料逻辑最完整。
**代价**：虽然成本在降低，但比起代码直接切分，它仍然需要支付额外的 API 费用，并消耗更多的预处理时间。它非常适合处理**结构极其复杂、对准确度要求极高**的专业文档（如司法解释、医学教材）。

## 十七、上下文感知分片

RAG（领域中一种非常前沿且高级的技术：**上下文感知分片（Context-Aware Chunking）**，有时也被称为“上下文检索（Contextual Retrieval）”。

### 1. 核心痛点：为什么传统的“分片”不够好？

在传统的 RAG 流程中，我们会把长文档切成很多小块。

- **问题：** 每一个小块在离开母体文档后，往往会失去**上下文**。
- **例子：** 假设一个分片内容是“他的主要贡献在于用户反馈循环”。如果单独检索到这一块，检索系统可能不知道“他”是谁，或者这是关于哪个产品的贡献。这会导致检索不准。

### 2. 第一页：上下文感知分片的工作原理

这一页展示了如何利用大模型在分片阶段“注入”灵魂：

- **处理流程：**输入：原始文档。**LLM 介入：** 在将文档切片的同时，把每一个分片连同整个文档（或摘要）一起发给 LLM。**生成背景：** LLM 会为每一个分片写一段极短的**背景说明**。
- **课件中的具体示例：**分片1（标题）：LLM 标注其背景为“应用发布标题”。**分片2（正文）：** LLM 标注其背景为“发布公告”。**分片3（教训）：** LLM 标注其背景为“关键开发教训的总结”。
- **结果：** 每一个 Chunk 不再只有原始文本，还附带了一个描述它“身份”的标签。

### 3. 成本、收益与整体架构

这一页深入探讨了这种技术在工程实践中的权衡：

- **昂贵的预处理：**这是该技术的最大代价。你需要为知识库里的**每一个分片**都调用一次 LLM。如果你的文档有 1 万个分片，就要支付 1 万次 API 调用费用。这发生在“摄取阶段”，即把数据存入数据库之前。
- **收益：更好的检索，不影响检索速度：**检索更准：在搜索时，系统会对比“问题”与“（分片文本 + 背景说明）”的整体含义。由于背景说明里包含了文档的核心信息，匹配度会大幅提升。**查询速度无影响：** 虽然预处理慢，但在用户真正提问时，数据库只是多搜索了一些文字，响应时间几乎不变。
- **双重帮助：**检索相关性： 帮助检索器精准找到那个看似普通但背景相关的分片。**生成响应：** 当这些带背景的分片发给最终的 LLM 时，模型能读懂这段话的来龙去脉，生成的答案更准确，不容易“断章取义”。

## 十八、如何选择分片

### 1. 固定宽度与递归字符切分

- **课件观点：** **优秀的默认选择**。
- **深度解析：**固定宽度： 简单粗暴，每隔 `N` 个字符或 Token 切一刀。**递归切分：** 比较聪明，先按段落切，太大的话再按句子切，再大就按单词切。它会尽量保持段落和句子的完整。
- **为什么是默认选择？** 它们**成本极低、速度极快**，不需要消耗额外的 API 费用。对于大多数通用任务（如处理普通文档、网页），这两种方法已经能提供非常不错的表现，是项目启动时的首选。

### 2. 语义分片与大模型分片

- **课件观点：** **性能更高，但也更复杂。需要实验来验证其价值。**
- **深度解析：**语义分片： 使用 Embedding 模型计算相邻句子间的相似度，当发现意思发生大转变（断层）时才切分。**大模型分片：** 直接让 LLM 阅读全文，并告诉它：“在逻辑结束的地方帮我切开”。
- **权衡：**性能： 因为是按“意思”切的，检索到的内容逻辑非常连贯，噪音少。**复杂性/成本：** 预处理数据时需要产生大量的向量计算或 LLM 调用费用，速度较慢。课件建议“通过实验验证”，意思是你需要对比它带来的**准确度提升**是否对得起它增加的**金钱和时间成本**。

### 3. 上下文感知分片

- **课件观点：** **能改进任何分片技术，但有一定成本。是一个非常好的“进阶改进”方向。**
- **深度解析：**核心逻辑：无论你用上述哪种方法切分，都会面临一个问题：分片孤立后丢失了母体信息。**做法：** 在每个小分片前面，额外加上一两句关于整个文档的摘要或背景信息。
- **为什么推荐作为“第一项改进”？**它能显著解决“断章取义”的问题。即使检索器只找回了一个很小的片段，由于它自带上下文，最终的 LLM 也能读懂这段话的来龙去脉。虽然增加了预处理成本和存储空间，但对召回率的提升通常立竿见影。

### 4.总结建议：你的实施路线图

1. **第一步（初级）：** 先用 **递归切分**。它能解决 80% 的问题，且基本免费。
2. **第二步（中级）：** 如果发现模型经常“看不懂检索到的资料”，尝试 **Context Aware Chunking**。在每个分片头上加一点文档背景，这是投入产出比最高的改进。
3. **第三步（高级）：** 如果对精度有极致追求，且预算充足，再考虑 **Semantic Chunking**。利用 AI 的能力在“语义断层”处精准下刀。

## 十九、查询重写

其核心思想是：**用户输入的原始问题往往是“糟糕”的，在交给检索器去找资料之前，先用大模型把它“整容”成更适合搜索的形式。**

### 1. 核心流程图

它展示了请求是如何被处理的：

- **凌乱的提问**：用户输入的原始问题。这类提问通常包含废话、口语、歧义，或者缺乏关键术语。
- **查询重写大模型**：这是一个中间步骤。系统先调用一次大模型（通常是一个反应快、成本低的小模型），其任务不是回答问题，而是“翻译”问题。
- **优化后的提问**：大模型处理后的结果。它更简洁、更专业、更像是一个搜索引擎的查询语句。
- **检索器**：拿着优化后的提问去向量数据库或知识库中找资料。

### 2. 为什么要重写？

课件右侧给出了一个针对**医学数据库**（症状与诊断关联）的具体指令示例，解释了重写的四个具体操作：

1. **澄清模糊短语**：用户可能会说“我肚子难受”，重写器会尝试引导或转化为更具体的位置描述（如“上腹部疼痛”）。
2. **使用医学术语**：数据库里的专业文档通常使用标准术语。用户说“心脏病发作”，重写器可能会加上“心肌梗死”，这样更容易匹配到专业文献。
3. **添加同义词**：增加搜索的“命中率”。例如用户搜“感冒”，重写器会加入“流感”、“上呼吸道感染”等词，扩大搜索范围。
4. **去除多余或干扰信息**：用户提问时常带感情色彩或背景故事（如“我昨天去公园散步，突然觉得...”），重写器会删掉“去公园散步”这类与检索无关的噪音，提高搜索的“信噪比”。

### 3. 为什么这项技术对 RAG 至关重要？

在 RAG 系统中，**“检索不到相关资料”**是导致系统失败的首要原因。

- **语义鸿沟**：用户的口语和数据库的专业文本之间存在巨大差异。
- **解决幻觉**：如果检索器因为关键词没对上而找回了错误的资料，最后大模型就会根据错误资料“一本正经地胡说八道”。

**总结：**
**查询重写**就像是在图书馆里，你跟一个资深图书管理员说你想找什么，他帮你把模糊的要求转换成精准的索书号，然后再去书架上找。这样做能显著提升 RAG 系统找资料的**准确率和召回率**。

### 4.核心要点解析

#### **要点一：针对重写器的提示词进行迭代**

- **含义**：重写器本身也是由 LLM 驱动的，因此它也需要一个“提示词”。
- **为什么重要**：你不能只是告诉 LLM“帮我重写这个提问”，你需要通过不断的测试（迭代），给它更精确的指令。
- **迭代的方向**：是否需要扩充同义词？是否需要把简短的口语扩充为详细的描述？是否需要针对特定的向量数据库优化关键词权重？
- **结论**：重写器的效果直接决定了检索的上限，因此需要像调优最终模型一样去精雕细琢重写器的 Prompt。

#### **要点二：收益巨大，足以抵消成本**

这是一个非常重要的工程决策参考。

- **成本**：**金钱成本**：多调用了一次 LLM API。**时间延迟**：用户需要多等一两秒才能看到答案（因为多跑了一个重写步骤）。
- **收益**：**召回率大幅提升**：能搜到原来搜不到的隐藏资料。**准确率提升**：过滤掉无关干扰，让检索到的内容更聚焦。**减少幻觉**：因为资料找对了，最终生成的答案更真实可靠。
- **结论**：在生产环境中，**检索失败**是 RAG 系统失败的首要原因。虽然查询重写增加了性能开销，但它带来的系统稳定性提高和质量飞跃，使得这些成本投入是完全值得的。

## 二十、命名实体识别

### 1. 什么是命名实体识别？

课件顶部的定义指出：NER 的作用是**识别**并**分类**查询语句中的特定类型信息。

简单来说，当用户输入一句话时，NER 就像一个“侦探”，能从普通的文字中把那些**专有名词**或**关键数据**拎出来，并告诉系统它们分别属于什么类别。

### 2. 五大常见实体类型

课件列举了 NER 经常识别的五种核心类别：

- **Places（地点）**：识别国家、城市、地标（如“北京”、“温哥华”）。
- **Dates（日期）**：识别具体的时间点、年份或时长（如“2024年”、“上周末”）。
- **People（人物）**：识别真实存在的个人姓名（如“泰勒·斯威夫特”）。
- **Characters（角色/虚拟人物）**：识别文学、影视作品中的角色名（如“福尔摩斯”）。
- **Organizations（组织机构）**：识别公司、学校、政府部门等（如“DeepLearning.AI”、“联合国”）。

### 3. NER 在 RAG 系统中的实际用途

课件提到它能实现“**更具针对性的搜索和过滤策略**”，这主要体现在以下两个方面：

#### **A. 结合元数据过滤**

这是 NER 最强大的应用场景。

- **例子**：用户问“**苹果公司**在**2023年**发布的财报内容是什么？”
- **NER 的动作**：提取出 苹果公司 -> **Organization**。提取出 2023年 -> **Date**。
- **系统的操作**：系统不再只是满大街找“苹果”这个词（可能会搜到水果），而是直接去数据库里执行**硬过滤**：WHERE organization = 'Apple' AND year = 2023。这样搜出来的资料百分之百准确。

#### **B. 引导查询重写**

- **逻辑**：如果 NER 识别出用户问的是一个特定的人物，重写器可以自动加上该人物的头衔或背景信息，帮助检索器在海量文档中更精准地定位。

## 二十一、**HyDE ** 

HyDE (Hypothetical Document Embeddings，假设性文档嵌入)的核心思想是：**与其用“问题”去搜“答案”，不如先让 AI 编一个“假答案”，然后用这个“假答案”去搜“真答案”。**

### 1.HyDE 的工作流程

这张图展示了 HyDE 是如何处理一个用户查询的：

1. **原始提问**：用户输入一个具体问题，例如：“我的左肩痛，大拇指和食指发麻，我该怎么办？”
2. **生成假设性文档**：系统并不直接拿着这个问题去数据库里搜。它先驱动一个大模型（LLM）针对这个问题写一篇**假设性的、理想的回答文档**。即使这个回答目前是“瞎编”的（可能包含幻觉），也没关系。
3. **向量化**：将这个由 LLM 生成的**假设性文档（假答案）**输入到嵌入模型中，转换成一个向量（例如图中显示的数字序列 [0, 5, 0, 4, ...]）。
4. **执行检索**：利用这个“假答案”的向量去向量数据库中寻找与之最相似的**真实文档**。

### 2.HyDE 的核心逻辑与优缺点

这一页总结了为什么我们要这么麻烦地多绕一步：

- **传统检索的痛点**：通常情况下，检索器是在用“短小的提问”匹配“长篇的文档”。提问和回答在**语义空间**里往往是不对等的（问题是疑问句，文档是陈述句）。
- **HyDE 的改进**：HyDE 将搜索变成了“文档匹配文档”。LLM 生成的假设性文档在语气、结构和用词上，与数据库里真正的知识文档更接近。这种**“语义对齐”**能显著提高检索的准确率。
- **实际考量**：**性能提升**：在处理复杂问题或知识库非常庞大时，HyDE 往往能找回更相关的资料。**代价**：**延迟 **：因为在搜索前必须先等 LLM 生成一段话，所以用户等待时间会变长。**成本**：每次搜索都要额外消耗一次 LLM 的 Token。

### 总结：为什么要用 HyDE？

**打个比方：**
如果你去图书馆找书。

- **普通 RAG**：你跟管理员说“我想知道怎么修电脑”，管理员根据这几个字帮你找。
- **HyDE**：你先写出一篇你心目中完美的“修电脑指南”草稿，然后跟管理员说“请帮我找一本跟我写的这篇内容最像的书”。

**HyDE 的神奇之处在于：** 哪怕 LLM 生成的“假答案”事实不准确，它所展现出的**语义特征（用哪些词、聊哪些概念）**也能极大地帮助检索器锁定正确的真实文档。

## 二十二、Bi-Encoder

### 1. 核心定义：独立向量化

- **双重路径**：在 Bi-Encoder 架构中，**提问**和**文档**是分别通过嵌入模型进行处理的。
- **语义映射**：模型将它们转化成空间中的坐标（向量）。在这个空间里，意思相近的内容会被放在相近的位置。

### 2. 效率的关键：文档向量预计算

这是 Bi-Encoder 能够处理海量数据的核心原因：

- **离线处理**：你可以提前将成千上万份文档全部转化成向量，并存入向量数据库中。这个过程不需要等待用户的提问。
- **实时响应**：当用户输入提问时，系统只需要对这**一个**提问进行向量化，然后直接去数据库里对比已经存好的文档向量。这极大地节省了在线计算时间。

### 3. 搜索方式：ANN 搜索

- **快速查找**：向量数据库使用 **ANN** 算法。
- **海里捞针**：它不需要挨个对比所有文档，而是能通过某种高效的索引（如 HNSW）在毫秒级时间内从几百万个向量中找出与提问向量“距离最近”的几个。

## 二十三、Cross-Encoder

### 1. 拼接文档与提示词 

- **操作：** 交叉编码器不会分别计算问题和文档的向量。相反，它将**用户的提问**和**候选文档**直接“拼接”在一起，中间通常用一个特殊的分隔符连成一长串文本。
- **关键点：** 文档是一个接一个地与提问进行拼接处理的。

### 2. 输入交叉编码器 

- **过程：** 这一长串拼接后的文本被整体输入到一个深度学习模型（通常是 BERT 等 Transformer 模型）中。
- **深度语境理解：** 这是交叉编码器的“杀手锏”。因为模型同时看到了问题和文档，模型内部的**注意力机制**可以让问题中的每一个字都与文档中的每一个字进行对比。
- **优势：** 它能理解极其微小的语义交互（例如：问题里的某个代词具体指向文档里的哪个细节），这种理解比单纯对比两个独立的向量（Bi-Encoder 做法）要深刻得多。

### 3. 生成相关性评分

- **结果：** 模型最后不输出向量，而是直接输出一个 **0 到 1 之间的数字**。
- **含义：** 这个分数直接代表了相关性。1.00 表示完全相关，0.00 表示毫无关系。
- **用途：** 系统根据这个分数对所有候选文档进行重新排序，把最精准的那个排在第一位。

### 4.总结

**交叉编码器就像是一个带着放大镜的审查员。** 虽然它不能像基础检索那样一次性扫描几百万篇文档，但它能对着你初筛出来的几十篇文档进行“逐字逐句”的审核，确保最后提供给大模型的内容是质量最高、最相关的。

## 二十四、Cross-Encoder在 RAG系统中的优点与缺点

### 1.优点 —— 为什么它“强”？

1. **几乎总是比 Bi-Encoder 提供更好的搜索结果 **原理： Bi-Encoder 是分别对问题和文档进行计算，信息在变成向量时会有损失。而 Cross-Encoder 把问题和文档**同时**输入模型，利用注意力机制捕捉两者之间极其微小的语义互动。**结果：** 它对相关性的判断极其精准，能分辨出那些关键词相似但逻辑不同的错误结果。
2. **非常适合改进其他搜索技术的结果**角色： 它通常不直接去数据库里“捞”数据，而是作为**“重排序器”**。**场景：** 你先用传统的关键词搜索（BM25）或 Bi-Encoder 快速捞出 100 个候选文档，然后用 Cross-Encoder 对这 100 个进行精细打分。

### 2.缺点—— 为什么它“慢”？

1. **在处理数百万或数十亿文档时扩展性极差**计算量爆炸： 如果你要在 100 万个文档里找答案，Cross-Encoder 需要把“你的问题”与“这 100 万个文档”**一一配对**后送入大模型计算 100 万次。**延迟：** 这在实时搜索中是完全无法接受的，用户可能要等几个小时才能拿到结果。
2. **无法进行预处理**对比： 在 Bi-Encoder 中，我们可以提前把文档转成向量存起来（预索引）。**限制：** Cross-Encoder 的得分取决于问题和文档的**组合**。你没法预测用户会问什么，所以你无法提前计算好任何分数。每一分都要在用户提问的那一刻“现算”。
3. **效率太低，无法作为默认（第一步）搜索技术 **因为它太重、太慢，它永远无法取代 Bi-Encoder 或传统索引作为“海选”工具。它只能处理少量的、经过初筛后的文档。

### 3.总结：RAG 中的最佳实践

这页课件告诉我们，在设计 RAG 系统时，通常采用 **“漏斗式”架构**：

- **第一步：海选**利用它们的**高效性**，从几百万文档中快速缩小范围到 50-100 个。
- **第二步：精选**利用它的**精准性**，对这 50-100 个文档进行深度比对，把最正确的 3-5 个发给大模型生成回答。

## 二十五、ColBERT

### 1. 弥合 Bi-Encoder 和 Cross-Encoder 的差距

- **背景回顾：**Bi-Encoder：速度极快，因为它将整个文档压缩成**一个**向量。但缺点是丢失了词与词之间的细节互动（精度较低）。**Cross-Encoder**：精度极高，因为它把问题和文档放在一起逐字对比。但缺点是速度极慢，无法处理大规模搜索。
- **ColBERT 的做法**：它允许我们像 Bi-Encoder 一样**提前计算**并存储文档的向量，但同时又像 Cross-Encoder 一样保留了**深层的文本互动**。

### 2. 每个 Token（词元）都有一个向量

这是 ColBERT 最显著的特征，也是它与普通向量检索最大的不同：

- **普通检索**：一句话生成一个向量（例如：[0.1, 0.5, ...]）。
- **ColBERT**：一句话中的**每一个单词**都会生成一个独立的向量。如图中所示：左边 Prompt 生成了一组向量，右边 Document 也生成了一组向量。这意味着模型保留了文档中每一个词的细微语境信息，而不是把它们揉成一团。

### 3. 延迟交互评分

这就是名字中“Late Interaction”的来源：

- **交互**：指问题和文档进行对比。
- **延迟**：指对比发生在向量生成**之后**，而不是在模型处理文本的过程中。
- **评分逻辑**：当用户输入 Prompt 时，Prompt 的每一个词向量都会去文档中寻找与其**最相似**的那个词向量。这种机制叫做 **MaxSim**（最大相似度）。最后，将所有 Prompt 词匹配到的最高分加起来，得出文档的总相关性得分。

### 总结：为什么要用它？

1. **更精准**：由于它在 Token 级别进行匹配，它能处理更复杂的查询，对关键词的捕捉比普通向量搜索更敏感。
2. **可扩展**：由于文档向量可以预先计算并存储（使用像 RAGatouille 这样的库），它依然可以实现毫秒级的快速检索。
3. **RAG 的利器**：在 RAG 系统中，ColBERT 常被用作高质量的**重排序器或直接作为多向量索引**，显著提升找回资料的相关性。

## 二十六、延迟交互及MaxSim

### 1.构建词对词的相似度矩阵

这张图展示了提问和文档之间 **Token 级别（单词级别）** 的互动。

- **纵轴**：用户的提问 “Great places to eat in New York”。
- **横轴**：知识库中的一段文档 “The cuisine of New York City comprises many...”。
- **表格中的数值**：代表提问中的每一个词与文档中的每一个词之间的**相似度分数**（通常是余弦相似度）。**数值越高（接近 1.0）**：表示两个词的意思越接近。**有趣的地方**：注意提问里的 eat（吃）和文档里的 cuisine（烹饪/菜系），它们的相似度是 **0.7**。虽然单词完全不同，但 ColBERT 的 Embedding 模型理解它们在语义上高度相关。**精确匹配**：提问里的 New 和文档里的 New 相似度高达 **0.9**。

### 2.计算 MaxSim 分数

这张图展示了 ColBERT 如何将这张复杂的表格浓缩成一个最终的“相关性总分”。

#### **计算逻辑：MaxSim **

ColBERT 的逻辑是：对于提问里的**每一个词**，我都去文档里找一个**跟它最像的词**，并记录下那个最高分。

1. **逐行寻找最大值（图中加粗的部分）**：Great 在文档里最像的词分值是 **0.3**。places 在文档里最像的词（City）分值是 **0.3**。to 在文档里最像的词分值是 **0.2**。eat 在文档里找到了 cuisine，分值是 **0.7**。in 在文档里找到了 of，分值是 **0.3**。New 在文档里找到了 New，分值是 **0.9**。York 在文档里找到了 York，分值是 **0.9**。
2. **求和**：
   将这些每行选出的最大值相加：
   `0.3+0.3+0.2+0.7+0.3+0.9+0.9=3.60.3+0.3+0.2+0.7+0.3+0.9+0.9=3.6`
3. **结果**：
   这个文档针对该提问的最终得分是 **3.6**。系统会根据这个分数对所有文档进行排名，分数越高，排名越靠前。

## 二十七、 ColBERT模型的优点与缺点

### 1. 优点

1. **兼具 Bi-encoder 的可扩展性和 Cross-encoder 的丰富交互性：**背景： 传统的 **Bi-encoder**（如普通向量检索）速度快但精度稍低，因为它将整个文档压缩成一个向量；**Cross-encoder** 精度极高但速度极慢，因为它要让问题和文档的每个词两两比对。**ColBERT 的优势：** 它采用了“延迟交互”机制。它像 Bi-encoder 一样，可以**预先计算并存储**文档的向量，因此具备在大规模数据中搜索的**可扩展性**。同时，它在检索时会对词对词的相似度进行计算，捕捉到了类似 Cross-encoder 的**深层语义交互**，使检索结果远比普通向量检索更精准。
2. **速度合理，支持实时场景：**虽然 ColBERT 的计算比简单的向量计算（点积/余弦）复杂一点，但它依然非常快。它可以在毫秒级内完成检索，因此完全可以用于**实时搜索**或对延迟要求较高的生产环境，而 Cross-encoder 通常只能用于对少量结果进行重排序。

### 2.缺点

1. **需要巨大的向量存储空间：**这是 ColBERT **最显著的缺点**。**对比：** 在普通向量检索中，一个文档块（Chunk）只对应 **1 个** 稠密向量。而在 ColBERT 中，文档里的**每一个词元（Token）**都要对应 **1 个** 稠密向量。**后果：** 假设一个文档有 200 个词，ColBERT 占用的存储空间就是普通向量检索的 **200 倍**。对于拥有数百万甚至数亿文档的系统来说，这会带来巨大的磁盘空间压力和内存成本。

## 二十八、重排序

### 1. RAG 整体工作流

这部分展示了重排序在系统中的位置：

- **User Query（用户查询）**：用户提出问题。
- **Initial Retrieval（初步检索）**：通常使用**混合检索**，即结合“向量搜索”和“关键词搜索”，从海量数据库中快速捞出前 50 到 100 个候选文档。
- **Reranking（重排序）**：这是本页的核心。在把资料喂给大模型之前，先进行一次精细化的筛选。
- **LLM Generation（大模型生成）**：大模型根据经过重排序后的最佳文档来生成回答。

### 2. 重排序引擎的工作原理

初步检索虽然快，但它对“相关性”的判断比较粗糙。重排序引擎会对初步筛选出的文档进行“深度体检”：

- **Top Ranked Documents from Hybrid Search**：输入的是初筛后的那一小堆文档。
- **Reranking Engine（重排序引擎）**：**Cross Encoder（交叉编码器）**：这是最常用的重排工具。它将用户问题和文档**拼在一起**输入模型，利用注意力机制逐字对比。它比初筛用的 Bi-Encoder 慢得多，但在小范围内（如前 50 篇）非常准。**LLM Scoring（LLM 打分）**：有时也可以直接调用一个更强的大模型（如 GPT-4）给这些文档的相关性打分，虽然成本高，但理解力极强。

### 3. 最终结果：优化文档顺序

- **Optimal Document Order（最优文档顺序）**：重排序后的结果。
- **直观表现**：原本在初筛中排在第 10 名的文档，可能经过重排后发现它才是真正的“满分答案”，于是被提到了第 1 名。
- **意义**：大模型的“注意力”是有局限的，且处理过长的上下文很贵。**把最精准的文档放在列表的最前面**，能显著减少大模型的幻觉，提高回答质量。

### 总结：为什么要重排序？

1. **快慢结合（精度权衡）**：初筛负责在百万级数据中“海选”，速度快；重排序负责在百级数据中“精选”，精度高。
2. **克服向量检索的盲点**：向量搜索（Bi-Encoder）有时会因为语义过于模糊而找错东西，重排序通过深度语义对比纠正这些错误。
3. **节省 Token / 提升响应质量**：LLM 通常对提示词**最开头**的内容最敏感。重排序确保 LLM 优先看到最正确的资料。



## 二十九、Cross-Encoder 作为重排序器RAG系统中的作用和原理

### 1. 它是如何工作的？

- **拼接：** 与普通的向量检索（Bi-Encoder）不同，Cross-Encoder 会将**提问**和**文档内容**强行拼接在一起，作为一个整体输入到模型中。
- **Cross-Encoder 模型：** 模型内部的“注意力机制”可以同时看到提问和文档中的每一个词。这让模型能够捕捉到两者之间极其细微的语义交互和逻辑关系。
- **相关性评分：** 模型最后不再生成向量，而是直接输出一个 **0.00 到 1.00** 之间的数值。1.00 代表文档与问题极度相关。0.00 代表毫无关系。

### 2. 右侧核心观点：为什么要用它？

使用 Cross-Encoder 的三个关键理由：

- **比 Bi-Encoder 效果更好，但速度更慢：**更好： 因为它能进行“逐字逐句”的深度对比，精度远高于只对比两个独立向量。**更慢：** 由于每对比一个文档都要跑一遍大模型，它无法像 Bi-Encoder 那样预先计算好向量，所以无法处理数百万级的大规模搜索。
- **实现“质量与时间”的平衡：**在实际应用中，我们**只在初始过滤之后**才使用它。**策略：** 先用快速的 Bi-Encoder（向量搜索）从海量库中捞出前 50 或 100 个候选文档（初筛），然后再用 Cross-Encoder 对这 50-100 个文档进行精准打分（重排）。这样既保证了速度，又保证了质量。
- **极小的延迟换取巨大的提升：**虽然增加了一个步骤，带来了微小的延迟（通常几十毫秒），但对于最终 RAG 生成的准确率提升通常是非常显著的。

## 三十、基于大语言模型的重排序评分

### 1. 核心流程：LLM 作为“裁判”

课件展示了系统如何判断一个文档与问题的相关性：

- **输入**：将**原始提问 **（例如：“加拿大的首都是哪里？”）和从数据库中初步找出的**文档** 一起喂给模型。
- **处理**：使用一个**经过微调的大语言模型**。这个模型经过专门训练，任务不是回答问题，而是作为一个“深度评审员”，分析文档内容是否真的能回答这个查询。
- **输出**：模型直接输出一个**相关性分数**。图中给出的示例是 **0.95**（表示极度相关）。

### 2. 为什么使用微调后的 LLM？

虽然普通的大模型也能做简单的相关性判断，但“微调后的 LLM”（如专门的 Reranker 模型）具有以下优势：

- **深层语义理解**：它能识别出复杂的逻辑关系、同义词替代以及微妙的上下文语境，而不仅仅是关键词匹配。
- **排除噪音**：它能敏锐地发现那些看起来包含关键词、但实际上逻辑牛头不对马嘴的“伪相关”文档。

基于 LLM 的计分功能强大但昂贵。与交叉编码器 (Cross-Encoder) 类似，它对于大规模检索来说太慢，最好用于初步过滤后的重排序。**LLM 评分是目前最准的检索技术，但它是“贵族化”的手段。** 在 RAG 系统中，它必须作为“漏斗”的最后一环，对初筛结果进行“精加工”，从而确保发给大模型的资料是万无一失的。

## 三十一、总结

### 1. 左侧：底层基础设施 

这部分解释了为了让检索“跑得动”，系统底层需要具备的能力：

- **近似最近邻**：**核心逻辑**：对比了“暴力搜索”。暴力搜索会对比库里所有数据，速度极慢；而 **ANN 算法**（如 HNSW）通过牺牲极小的精度换取了巨大的速度提升。**重要性**：它是 RAG 能够处理百万甚至亿级文档量的前提。
- **向量数据库**：**核心逻辑**：这是一种专门设计的数据库，专门用于存储**高维向量**，并内置了上述的 **ANN 搜索**功能。**重要性**：它是 RAG 系统的“心脏”，负责高效地存取和检索知识。

### 2. 右侧：RAG 核心技术 

这部分列出了在数据处理和检索流程中，开发者可以操作的三个关键环节：

- **分片**：**作用**：将长文档切成小块。**关键点**：切片的大小和重叠度直接影响检索的精度。太大会引入杂音，太小会丢失语境。
- **查询解析 (Query Parsing)**：**作用**：在搜索之前先理解或转换用户的提问。**手段**：包括之前学过的**查询重写**、**命名实体识别（NER）**以及 **HyDE（假设性文档嵌入）**。**目的**：解决用户问得不清楚、口语化，导致搜不到正确资料的问题。
- **重排序**：**作用**：对检索回来的初步结果进行二次精选。**手段**：使用 **Cross-Encoder**、**ColBERT** 或专门的重排模型。**目的**：确保发给大模型的前几条资料是最精准、最相关的，从而减少幻觉，提升回答质量。
