
## 一、检索器（Retriever）

### 1. 管理可信信息的知识库

- **含义：** 检索器并不产生新知识，它负责连接和管理一个**预先构建好的知识库**。
- **关键点：** 这个知识库通常包含企业文档、PDF、数据库、实时新闻等**受信任的原始资料**。由于这些资料是经过审核或来源可靠的，因此它为系统提供了“事实真相”的源头。

### 2. 寻找最相关的信息并与 LLM 共享

- **含义：** 这是检索器最核心的工作逻辑，即**“过滤”与“桥接”**。
- **流程：** 当用户提出问题时，检索器会像搜索引擎一样，在浩如烟海的知识库中进行“搜索”。它通过算法（如向量匹配或关键词匹配）筛选出最相关的几条信息。最后，它把这些筛选出来的“干货”打包发送给大模型。
- **比喻：** 就像在一场考试中，你不需要背下整本教科书，检索器会在你遇到难题时，精准地帮你翻到教科书对应的那一页。

### 3. 改善生成质量

- **含义：** 这是检索器的最终目的。
- **效果：** 通过给大模型提供具体的参考资料，大模型生成回答时就不再是“盲目猜测”，而是“看图说话”。
- **结果：** 这样生成的回答会更加**准确**（减少幻觉）、**专业**（包含特定领域的细节）且**具有时效性**。

## 二、RAG的优势

### 1. 注入缺失的知识

- **含义：** 大模型的知识来自于训练数据，但训练是有截止日期的（例如它可能不知道昨天的天气或公司昨晚新发布的政策）。
- **优势：** RAG 允许我们将大模型从未见过的信息（如最新的政策文件、实时更新的内部文档）直接“喂”给模型，填补其知识盲区。

### 2. 减少幻觉

- **含义：** “幻觉”是指大模型在不知道答案时，会一本正经地胡说八道。
- **优势：** RAG 让模型基于检索到的**事实内容**来回答（即“看图说话”）。有了真实的背景资料作为依据，模型编造事实的可能性会大幅降低。

### 3. 保持模型实时更新

- **含义：** 重新训练一个大模型非常昂贵且耗时。
- **优势：** 我们不需要动大模型本身，只需要更新模型背后的知识库。这样，即使模型是去年训练的，通过 RAG 检索最新的知识库，它依然能回答今天的问题。

### 4. 支持引用来源

- **含义：** 纯大模型的回答是一个“黑盒”，你不知道它从哪得出的结论。
- **优势：** RAG 可以显示回答是基于哪些具体文档生成的（就像论文后的参考文献）。这让答案可验证、可追踪，极大增强了答案的公信力。

### 5. 让模型专注于“生成”

- **含义：** 术业有专攻。
- **优势：** 这种架构进行了分工。
  **检索器：** 专门负责在海量数据中“找事实”。
  **大模型：** 不需要背书，只需要发挥其强大的语言组织能力，把检索到的事实写成流畅的回复。

## 三、Token（词元）

### 1. 什么是 Token？

课件通过几个要点定义了 Token 的特性：

- **单词的片段：** Token 并不总是等于一个完整的单词，它可以是单词的一部分。
- **简单词通常对应单个 Token：** 比如图中右侧的 **London** 和 **door**，它们很常用，所以模型直接给它们分配了一个独立的 Token。
- **复杂词/长词使用多个 Token：** unhappy 被切分成了两个：un + happy。**programmatically** 这种长词被切分成了三个：program + m + atically。
  *逻辑：* 这样做可以让模型通过组合基础模块（如前缀 un-）来理解它从未见过的生僻词。
- **标点符号也是 Token：** 逗号、感叹号、空格等都会被计入 Token。例如图中底部的 **"Completely, I agree!"**，逗号和感叹号都是独立的 Token。

### 2. LLM 的词表

- **规模：** 大多数模型的词表包含 **10,000 到 100,000 个** 预定义的 Token。
- **优势：** 这个规模是一种平衡。它允许模型用最少的 Token 组合来表示世界上几乎任何单词（包括各种语言、代码和新造词）。

### 3. 视觉示例分析（图中的颜色块）

- **顶部的句子：** "What a beautiful day, the sun is shining in the sky"。
  每一个颜色方块代表一个 Token。注意：有些方块包含了单词前面的空格（例如 shining）。在 Token 化过程中，空格通常会附着在后面的词上。
- **底部的词组：** "Completely, I agree!"
  紫色块：Completely
  黑色/暗绿块：, (逗号)
  黄色块：I (空格+I)
  红色块：agree (空格+agree)
  紫色块：! (感叹号)

### 4. 为什么要理解 Token？

1. **计费与限制：** 所有的 API 调用（如 OpenAI, Together.ai）都是按 Token 数量计费的，而不是按字数。
2. **窗口限制：** 之前课件提到的“上下文窗口限制”也是以 Token 为单位计算的。
3. **效率：** 了解 Token 如何构成，有助于优化提示词。例如，如果提示词中有很多生僻词或复杂的标点符号，消耗的 Token 会比普通文本多得多。

## 四、自回归生成

### 1. 自我影响

这意味着模型生成文本是一个“循环”过程。模型每生成一个新词元，这个新词就会立即变成下一次预测的“输入”的一部分。
*公式化理解：* 输入 A -> 得到 B；接着输入 A+B -> 得到 C；再输入 A+B+C -> 得到 D。模型不断地“吃掉”自己刚刚写出的内容。

### 2. 新词元在旧词元的语境下是有意义的

模型不仅仅是随机蹦单词，它会严格参考之前所有的词。比如图中预测出“shining”之后，后面的“in the sky”必须在“sun is shining”这个语境下合乎逻辑。

### 3. 运行相同的提示词会导致不同的补全

这是因为模型不是死板的“查表”，而是**基于概率采样**。对于同一个开头，模型计算出很多可能的后续，并随机选择一个，所以才有了图中分叉的两种结果。

## 五、为什么要“在概率分布中随机选择”而不是“只选最高概率”

### 1. 避免陷入“死循环”、重复和呆板

如果模型总是选择概率最高的词，它非常容易陷入逻辑死循环。

- **例子：** 模型可能会生成这样的句子：“我喜欢吃苹果，因为苹果很好吃，所以我喜欢吃苹果，因为苹果很好吃……”
- **原理：** 概率最高的词往往是那些最稳妥、最常见的词。一旦进入某种常见的句式，最高概率的选择往往会指向已经出现过的词，导致模型不停地原地打转。

### 2. 增加“创造性”和“自然感”

人类说话并不是在做统计学上的“最优选择”。

- **例子：** 当我说“今天天气真...”时，概率最高的词可能是“好”。但如果模型永远只说“好”，它就会变得极其无聊。
- **原理：** 通过在概率前几名的词中随机选择（比如从“好”、“不错”、“晴朗”、“糟糕”里选），模型能生成更多样化、更生动、更像人类的回答。

### 3. 防止生成质量退化

**局部最优并不等于全局最优。**

- **比喻：** 就像下棋，你每一步都吃掉对方目前最值钱的子（局部最优），但可能因此走进了对方的陷阱，导致全局满盘皆输。
- **原理：** 在生成文本时，当前概率最高的词可能会导向一个非常难接下去的结尾，导致后面的句子逻辑崩溃。而选择一个概率第二高但“潜力更好”的词，往往能生成质量更高的长段落。

### 4. 我们可以通过“温度（Temperature）”来控制这种随机性

在调用 LLM 的 API 时（比如你之前代码里的参数），有一个参数叫 **temperature**，它就是用来调节这个“随机程度”的：

- **Temperature = 0：** 这就是你说的，**几乎只选概率最高的那个词**。适合需要极度严谨的任务，如数学计算、写代码。
- **Temperature = 0.7 - 1.0：** 这是默认状态，模型会在高概率的几个词中随机跳选。适合聊天、写故事、创意写作。
- **Temperature > 1.0：** 模型会变得非常“疯狂”，甚至选择那些概率很低的词，回答会变得语无伦次。

### 5. 采样策略（Top-K 和 Top-P）

为了防止模型随机选到一个完全离谱的词（比如概率只有 0.00001% 的词），科学家发明了两种限制方法：

- **Top-K：** 只从概率最高的前 K 个词里随机选。
- **Top-P (Nucleus Sampling)：** 只从累计概率达到 P（比如 90%）的词群里随机选。

### 总结

- **只选概率最高**（贪婪搜索）：回答准确但死板，容易重复，缺乏灵性。
- **基于概率随机选**（采样生成）：回答自然、多样、更像人，且能处理复杂的长文本。

## 六、为什么大语言模型会产生幻觉

### 1. LLM 生成的是“概率性”的词序列

- **课件原话：** LLMs just reproduce statistical patterns from their training data.（LLM 只是在重现其训练数据中的统计模式。）
- **深度解释：** 模型本质上是一个极其复杂的“概率预测器”。当你给它一个开头时，它并不会去思考“什么是真理”，而是通过数学计算，找出在它的训练库（海量互联网文本、书籍等）中，后面接什么词的**概率最高**。它是在模仿人类说话的**统计规律**，而不是在进行逻辑推理。

### 2. 知识空白导致不准确的回答

- **课件原话：** Responses can "sound right" but aren't true.（回答听起来“很有道理”，但并非事实。）
- **深度解释：** 模型在训练时如果没见过某个知识点，或者对该知识点的记忆模糊，它**不会像人类一样轻易承认“我不知道”**。大模型的训练机制，宁愿去猜一个答案，也不会说自己不知道，因为有可能猜对从而提高准确率。
- **“听起来很对”：** 由于模型完美掌握了语言的语法和语气，它会用非常自信、专业的口吻编造一个答案。这就像一个考试时完全不会做题的学生，虽然内容是瞎编的，但字迹工整、结构标准，极具误导性。

### 3. 真实 `≠\\=` 高概率

- **课件原话：** LLMs are designed to generate "probable" text, not truthful text.（LLM 的设计初衷是生成“高概率”文本，而非“真实”文本。）
- **深度解释：** 这是最关键的一点。大模型的目标函数是最小化“预测下一个词”的误差。
  **例子：** 如果训练数据中流传着很多谣言，或者某个错误的说法出现的次数足够多，那么对于模型来说，这个**错误的说法**就是“高概率”的。
  **结论：** 模型追求的是文本的“连贯性”和“自然度”，它缺乏一个内置的“事实核查器”来区分什么是概率上的自然连接，什么是客观世界的真实事实。

**解决方法：** RAG 技术或者优化大模型训练的激励机制（不要只奖励大模型猜对了答案，对于大模型诚实地回答不知道时，也要给予相应奖励）

## 七、为什么我们不干脆把“所有的资料”都直接塞进提示词里

### 1. 更高的计算成本

即使大模型能读完所有的资料，代价也非常昂贵。

- **提示词越长，计算量越大**
  大模型处理文本的开销不是恒定的。提示词每增加一倍，模型处理它所需的计算资源（GPU 算力和电力）通常会显著增加。
- **对每一个词元进行复杂的扫描：**
  这是由 Transformer 架构的“自注意力机制”决定的。模型必须计算当前词与之前**所有词**之间的关系。
- **生成每个新词前都要扫描一次：**
  这是最耗时的地方。模型每写出一个新词，它都要把前面的长篇大论重新“温习”一遍，以确保逻辑连贯。如果提示词有几万字，生成每一个字都会变得非常缓慢且昂贵。

### 2. 上下文窗口限制

大模型的“内存”是有上限的，就像一个人的短时记忆长度是有限的。

- **最终你会触及上限：**
  每个模型都有一个预设的**最大承载量（Context Window）**。一旦资料量超过这个上限，模型就会“溢出”，后面的内容它根本看不见，或者会丢掉前面的内容。
- **小型模型：只有几千个词元：**
  许多轻量化或早期的模型（如传统的 Llama 或 GPT-3.5 早期版），只能处理大约 4000 到 16000 个词元（约合几千个单词）。这点容量连一本厚点的书都放不下。
- **大型模型：数百万个：**
  虽然像 Gemini 1.5 Pro 或 Claude 3 这样尖端模型已经支持百万级别的窗口，但即便如此，资料库（比如公司的全部技术文档或整个维基百科）依然远远超过这个量级。

## 八、检索器的权衡与取舍

### 1. 核心矛盾：相关性 vs 不相关性

- **目标：** 检索器的理想状态是只返回**高度相关**的文档，并完全排除**不相关**的文档。
- **挑战：** 在现实中，检索器很难精准地区分这两者。

### 2. 两个极端的权衡

课件对比了两种极端的做法及其弊端：

- **极端 A：返回所有文档后果：** 会混入大量无关的废话（“海量不相关文档”）。
  **代价：** 浪费大模型的**上下文窗口**，增加计算成本，并可能干扰模型的判断。
- **极端 B：只返回排名第一的文档：** 极易遗漏关键信息。
  **原因：** 最完美的答案可能分散在多篇文档中，或者检索算法没能把最正确的答案排在第一位。

### 3. 现实的无奈：没有完美的方案

- **检索器的局限性：** 现有的检索技术（如向量搜索）无法做到 100% 完美的排名。排名第 2 或第 3 的文档往往包含着排名第 1 的文档所缺失的重要补充。

### 4. 解决对策：监控与实验

- **动态调整：** 既然没有标准答案，开发者必须通过不断的实验来寻找“甜蜜点”（Sweet Spot）。
- **优化手段：** 比如调整 top_k 的值（即决定给 LLM 看前几篇文档），并观察哪种设置下模型的回答最准确、最省钱。

## 九、实际落地中的实现方式

### 1. 关系型数据库

- **课件描述：** "Already widely adopted"（已被广泛采用）。
- **特点：** 这是传统的数据库（如 MySQL, PostgreSQL, SQL Server）。数据以表格、行、列的形式存储。通过特定的关键词或精确的条件（SQL 语句）进行查询。
- **在 RAG 中的角色：** 虽然它不是为 RAG 量身定做的，但因为企业的大部分数据（如产品清单、订单记录、用户信息）已经存在这里，所以它仍然是 RAG 的重要数据源。
  **局限性：** 很难进行“语义搜索”。例如，你搜“开心”，它可能找不到包含“愉快”的内容，因为它只认精确的关键词匹配。

### 2. 向量数据库

- **课件描述：** "Specialized for retrieval in a RAG system"（专门为 RAG 系统中的检索而设计）。
- **特点：** 这是 RAG 系统的“秘密武器”。它不存储简单的文本，而是存储由模型生成的**向量（Vectors）**，即代表文本含义的一串数字。它使用“相似度搜索”而不是关键词匹配。
- **在 RAG 中的角色：** 语义理解：它能理解文字背后的意思。如果你搜“很棒的运动鞋”，它能帮你找回描述里写着“卓越的跑步装备”的文档，即便单词完全不同。
  **高效：** 专门针对大规模向量数据的快速对比进行了优化，能在几百万条数据中瞬间找到最相关的几条给 LLM。

## 核心概念

### 1. RAG 是 LLM 与知识库的“配对”

- **解释：** RAG 不仅仅是一个大模型，它是一套系统。它将**大模型的生成能力**与**外部知识库的存储能力**结合在一起。
- **比喻：** LLM 是一个“聪明的大脑”，而知识库是一本“百科全书”。RAG 就是让这个大脑在回答问题前先去翻阅这本百科全书。

### 2. 解决数据缺失问题

- **背景：** 大语言模型是在公开的数据集上训练的，因此它存在三个局限性。
  **私有数据：** 比如你公司的内部合同、个人笔记。
  **最新数据：** 比如今天早上刚刚发生的突发新闻。
  **高度专业化数据：** 比如某个极其冷门领域的科研资料。
- **结论：** 因为这些信息不在 LLM 的原始训练数据里，所以模型无法直接回答。RAG 的存在就是为了把这些“模型不知道的信息”补齐。

### 3. 检索并构建增强提示词

- **流程：** 这是 RAG 的具体动作。
  **检索：** 当用户提问时，系统中的“检索器”会去知识库里搜索相关的文档。
  **增强：** 系统将搜索到的文档内容与用户的原始问题拼接到一起，形成一个内容更丰富的**增强型提示词**。
- **意义：** 这样模型拿到的不再是一个孤零零的问题，而是一个“带参考答案的问题”。

### 4. 模型基于检索信息回答

- **核心目标：** **锚定/对齐**。
- **解释：** LLM 接收到增强提示词后，不再仅凭记忆去“猜”答案，而是被要求**根据提供的参考资料**来组织语言。
- **结果：** 这大大降低了模型胡编乱造（幻觉）的可能性，确保回答是有据可查的。
